The Problem

LLM costs spiral out of control fast

Most teams pick one model and send every request to it — paying premium prices for tasks that a fraction of the cost could handle just as well.

💸

Uniform pricing, variable complexity

Asking "What's the capital of France?" costs the same as "Debug this 200-line Python script" when you hard-code a single model. That's wasteful.

🔀

Routing logic buried in app code

Teams that do try to optimise embed routing logic directly in their applications, coupling cost decisions to business code across every service.

📉

No visibility into per-request cost

Without a centralised gateway, there's no single place to observe, audit, or enforce cost policies — making cost control nearly impossible at scale.

The Solution

Kong classifies every prompt before it reaches any LLM

A single Lua plugin inside Kong Gateway inspects the request body, scores the prompt's complexity, and silently rewrites which model handles it. Your frontend sends one standard OpenAI-compatible request — Kong does the rest.

1

Browser sends a standard chat request

The frontend posts to POST /v1/chat/completions on Kong (port 8000) with a normal OpenAI-format JSON body. It has no idea which model will respond.

Content-Type: application/json
2

Kong's pre-function Lua plugin classifies the prompt

Before forwarding the request, Kong runs a small Lua script that:

  • Buffers and parses the JSON request body
  • Extracts the user's message text
  • Matches it against a keyword list (debug, code, algorithm, analyze…)
  • Falls back to length-based scoring (> 400 chars = complex)
X-LLM-Tier: simple | complex X-Model: gpt-4o-mini | claude-3-5-sonnet
3

Kong proxies to the Smart Router service

Kong forwards the enriched request (with the classification headers it just set) to the internal Smart Router microservice. Kong also enforces rate limiting (30 req/min) and handles CORS for the browser.

Rate-Limiting: 30/min CORS: *
4

Smart Router calls the right LLM API

The FastAPI Smart Router reads X-LLM-Tier from the Kong headers and dispatches to the appropriate backend:

  • Simple → OpenAI gpt-4o-mini
  • Complex → Anthropic claude-3-5-sonnet-20241022
OpenAI API / Anthropic API
5

Response returns with cost metadata

The response travels back through Kong to the browser in standard OpenAI format, augmented with an x_smart_cost field showing the model used, token counts, actual cost, and how much was saved vs. always using the expensive model.

x_smart_cost: { tier, cost_usd, savings_usd, savings_pct }

Components

What's inside this PoC

Four Docker containers, one docker compose up.

K

Kong Gateway :8000 (proxy) · :8001 (admin)

Runs in DB-less mode using a declarative kong/kong.yml config. Hosts the pre-function Lua classifier plugin, the CORS plugin, and rate limiting. No database required — all configuration is version-controlled YAML.

R

Smart Router (FastAPI) :8080

A lightweight Python service that reads the X-LLM-Tier and X-Model headers set by Kong, then calls the appropriate LLM API (OpenAI or Anthropic). In MOCK_MODE (default) it returns realistic simulated responses with full cost breakdowns — no API keys needed.

UI

Chat Frontend (Nginx) :3000

A single-page chat interface that sends all requests to Kong in standard OpenAI format. Displays a colour-coded model badge on each response (green = gpt-4o-mini, amber = claude-3-5-sonnet) and a live cost analytics sidebar showing savings per request and cumulative totals.

L

Lua Classifier (Kong pre-function plugin)

Embedded directly in kong/kong.yml. Runs in the Kong request access phase — before the request leaves the gateway. Reads the raw request body via ngx.req.read_body(), parses JSON, scores the prompt, and sets routing headers. Total overhead: sub-millisecond.

Key Code

The Lua classifier (inside Kong)

This snippet lives inside kong/kong.yml as the pre-function plugin's access phase script. Kong executes it on every inbound request. 

-- 1. Buffer the request body (required to read it in Kong)
ngx.req.read_body()
local body_data = ngx.req.get_body_data()

-- 2. Parse JSON and extract the user's prompt text
local ok, parsed = pcall(require("cjson").decode, body_data)
local prompt = ""
for _, msg in ipairs(parsed.messages) do
  if msg.role == "user" then prompt = prompt .. msg.content .. " " end
end

-- 3. Keyword-based complexity classifier
local complex_signals = {
  "debug", "bug", "error", "fix", "code", "script", "function",
  "implement", "refactor", "algorithm", "optimize", "architecture",
  "explain how", "how does", "analyze", "performance", "security",
}
local t = string.lower(prompt)
for _, signal in ipairs(complex_signals) do
  if string.find(t, signal, 1, true) then
    kong.service.request.set_header("X-LLM-Tier", "complex")
    kong.service.request.set_header("X-Model", "claude-3-5-sonnet-20241022")
    return
  end
end

-- 4. Default: lightweight → cheap model
kong.service.request.set_header("X-LLM-Tier", "simple")
kong.service.request.set_header("X-Model", "gpt-4o-mini")
Quick Start

Up and running in under 2 minutes

No API keys required in mock mode.

1

Clone the repo

git clone https://github.com/rifaterdemsahin/kong-gateway.git
cd kong-gateway
2

Start the stack (Docker required)

./setup.sh
# or: docker compose up --build
3

Open the chat UI

open http://localhost:3000
4

Use real LLM APIs (optional)

# Edit .env:
MOCK_MODE=false
OPENAI_API_KEY=sk-...
ANTHROPIC_API_KEY=sk-ant-...

docker compose restart smart-router
