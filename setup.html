Fly.io vs Azure — Side by Side

Full step-by-step deployment for both platforms, with a real cost breakdown and maintenance comparison so you can pick the right home for your Kong gateway.

What You're Deploying

Three services, one stack

The Kong Smart-Cost stack consists of three containers that need to run and talk to each other. Both platforms handle this, but in very different ways.

Services
kong (kong:3.9)  — DB-less gateway, proxy :8000, admin :8001
smart-router (custom FastAPI)  — LLM proxy :8080
frontend (nginx:alpine)  — static chat UI :80
Kong and smart-router must reach each other. Only Kong proxy (:8000) and frontend (:80) need public ingress.

Fly.io — Step-by-Step Deployment

Prerequisites

You need a Fly.io account (free tier works) and Docker installed locally.

# Install Fly CLI
curl -L https://fly.io/install.sh | sh

# Authenticate
fly auth login

# Confirm it works
fly version
1

Create a Fly organisation and set secrets

All three services share the same org. Secrets are set once per app — Fly injects them as environment variables at runtime. Your API keys never appear in code or config files.

# Set API key secrets (replace with real values)
fly secrets set OPENAI_API_KEY="sk-..." \
               ANTHROPIC_API_KEY="sk-ant-..." \
               MOCK_MODE="false" \
               --app smart-router   # (create the app first in step 3)
2

Deploy the Smart Router

The Smart Router is a custom Docker image. Fly builds it from your smart-router/ directory and deploys it to a private internal address — Kong talks to it over Fly's private 6PN network, so it never needs a public port.

smart-router/fly.toml
app = "smart-cost-router"
primary_region = "lhr"   # London — pick your closest region

[build]
  dockerfile = "Dockerfile"

[http_service]
  internal_port = 8080
  auto_stop_machines  = true   # scale to zero when idle
  auto_start_machines = true
  min_machines_running = 0
  force_https = true

[[vm]]
  size = "shared-cpu-1x"
  memory = "256mb"

# Deploy from inside the smart-router/ directory
cd smart-router
fly launch --name smart-cost-router --no-deploy
fly secrets set OPENAI_API_KEY="sk-..." ANTHROPIC_API_KEY="sk-ant-..." MOCK_MODE="false"
fly deploy

# Note the internal hostname Fly assigns:
# smart-cost-router.internal
3

Deploy Kong Gateway

Kong runs in DB-less mode. Mount your kong/kong.yml via a Fly volume, and point Kong's upstream host at smart-cost-router.internal — the private DNS name Fly gives your router app. Update kong/kong.yml to use this hostname before deploying.

kong/fly.toml
app = "smart-cost-kong"
primary_region = "lhr"

[build]
  image = "kong:3.9"

[env]
  KONG_DATABASE          = "off"
  KONG_DECLARATIVE_CONFIG = "/kong/declarative/kong.yml"
  KONG_PROXY_LISTEN      = "0.0.0.0:8000"
  KONG_ADMIN_LISTEN      = "127.0.0.1:8001"   # admin NOT public
  KONG_LOG_LEVEL         = "info"
  KONG_PLUGINS           = "bundled"

[[mounts]]
  source      = "kong_config"
  destination = "/kong/declarative"

[http_service]
  internal_port = 8000
  force_https   = true
  auto_stop_machines  = false   # keep Kong warm
  min_machines_running = 1

[[vm]]
  size   = "shared-cpu-1x"
  memory = "512mb"

# Edit kong/kong.yml: change host from "smart-router" to
# "smart-cost-router.internal"
sed -i 's/host: smart-router/host: smart-cost-router.internal/' kong/kong.yml

cd kong
fly launch --name smart-cost-kong --no-deploy
fly volumes create kong_config --size 1    # 1 GB volume for config

# Copy kong.yml into the volume via a one-off machine
fly machine run kong:3.9 \
  --volume kong_config:/kong/declarative \
  --command "sh -c 'cat > /kong/declarative/kong.yml'" \
  --stdin < kong.yml

fly deploy
4

Deploy the Frontend

Update frontend/index.html to point to your Kong app's public URL (e.g. https://smart-cost-kong.fly.dev) instead of localhost:8000 before deploying.

frontend/fly.toml
app = "smart-cost-ui"
primary_region = "lhr"

[build]
  image = "nginx:alpine"

[http_service]
  internal_port = 80
  force_https   = true
  auto_stop_machines = true
  min_machines_running = 0

[[vm]]
  size   = "shared-cpu-1x"
  memory = "128mb"

# Replace localhost:8000 with your Kong Fly URL
KONG_URL="https://smart-cost-kong.fly.dev"
sed -i "s|http://localhost:8000|$KONG_URL|g" frontend/index.html

cd frontend
fly launch --name smart-cost-ui --no-deploy
fly deploy
5

Add a custom domain (optional)

# Add your domain to Kong (the public-facing app)
fly certs add api.yourdomain.com --app smart-cost-kong

# Point your DNS CNAME to:
# smart-cost-kong.fly.dev

# Fly auto-provisions and renews TLS via Let's Encrypt
fly certs show api.yourdomain.com --app smart-cost-kong
6

Verify everything works

# Check all apps are running
fly status --app smart-cost-router
fly status --app smart-cost-kong
fly status --app smart-cost-ui

# Test the gateway end-to-end
curl -X POST https://smart-cost-kong.fly.dev/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"auto","messages":[{"role":"user","content":"Summarize Kong Gateway in one sentence."}]}'

# Tail logs from Kong to see classifier decisions
fly logs --app smart-cost-kong
Fly.io deployment complete. Total time: ~20 minutes.
Azure — Step-by-Step Deployment

Prerequisites

Azure account, Azure CLI installed, and Docker installed locally.

# Install Azure CLI (macOS)
brew update && brew install azure-cli

# Log in
az login

# Set variables used throughout — edit these
RG="smart-cost-rg"
LOCATION="uksouth"
ACR="smartcostacr"          # must be globally unique, lowercase
ENV="smart-cost-env"
1

Create Resource Group and Container Registry

All Azure resources live in a resource group. The Container Registry stores your custom Docker images — Kong uses the official image directly, so only the smart-router needs to be pushed here.

# Resource group
az group create --name $RG --location $LOCATION

# Azure Container Registry
az acr create \
  --resource-group $RG \
  --name $ACR \
  --sku Basic \
  --admin-enabled true

# Build and push the smart-router image
az acr build \
  --registry $ACR \
  --image smart-router:latest \
  ./smart-router
2

Store secrets in Azure Key Vault

Key Vault keeps API keys out of your config files and provides a full audit log of every secret access. Container Apps can pull secrets directly from Key Vault using managed identity — no credentials in environment variables.

# Create Key Vault
az keyvault create \
  --name smart-cost-kv \
  --resource-group $RG \
  --location $LOCATION

# Store your API keys
az keyvault secret set --vault-name smart-cost-kv \
  --name openai-api-key     --value "sk-..."
az keyvault secret set --vault-name smart-cost-kv \
  --name anthropic-api-key  --value "sk-ant-..."
3

Create a Container Apps Environment

The environment is the shared VNet boundary where all three containers live. Services within the same environment can reach each other by name without public exposure. A Log Analytics workspace is attached for centralised logging.

# Log Analytics workspace (for observability)
az monitor log-analytics workspace create \
  --resource-group $RG \
  --workspace-name smart-cost-logs

LAWS_ID=$(az monitor log-analytics workspace show \
  --resource-group $RG \
  --workspace-name smart-cost-logs \
  --query customerId -o tsv)

LAWS_KEY=$(az monitor log-analytics workspace get-shared-keys \
  --resource-group $RG \
  --workspace-name smart-cost-logs \
  --query primarySharedKey -o tsv)

# Container Apps environment
az containerapp env create \
  --name $ENV \
  --resource-group $RG \
  --location $LOCATION \
  --logs-workspace-id $LAWS_ID \
  --logs-workspace-key $LAWS_KEY
4

Deploy the Smart Router

Deploy as an internal-only Container App — no public ingress, only reachable by other apps in the same environment. Scale to 0 when idle to eliminate cost during quiet hours.

ACR_LOGIN=$(az acr credential show --name $ACR --query username -o tsv)
ACR_PASS=$(az acr credential show  --name $ACR --query "passwords[0].value" -o tsv)

az containerapp create \
  --name smart-router \
  --resource-group $RG \
  --environment $ENV \
  --image ${ACR}.azurecr.io/smart-router:latest \
  --registry-server ${ACR}.azurecr.io \
  --registry-username $ACR_LOGIN \
  --registry-password $ACR_PASS \
  --target-port 8080 \
  --ingress internal \          # NOT public — only reachable within env
  --min-replicas 0 \
  --max-replicas 3 \
  --cpu 0.25 --memory 0.5Gi \
  --env-vars \
    OPENAI_API_KEY=secretref:openai-api-key \
    ANTHROPIC_API_KEY=secretref:anthropic-api-key \
    MOCK_MODE=false \
  --secrets \
    openai-api-key=keyvaultref:https://smart-cost-kv.vault.azure.net/secrets/openai-api-key,identityref:system \
    anthropic-api-key=keyvaultref:https://smart-cost-kv.vault.azure.net/secrets/anthropic-api-key,identityref:system
5

Deploy Kong Gateway

Kong uses the official image and reads config from an Azure File Share mounted as a volume. The upstream host in kong.yml becomes the Container Apps internal FQDN for the smart-router.

# Create Azure File Share for kong.yml
SA=smartcostsa$RANDOM
az storage account create --name $SA --resource-group $RG --sku Standard_LRS
az storage share create  --name kong-config --account-name $SA

# Upload kong.yml (edit host to smart-router's internal FQDN first)
# Internal FQDN pattern: <appname>.internal.<env-default-domain>
ROUTER_FQDN=$(az containerapp show --name smart-router --resource-group $RG \
  --query properties.configuration.ingress.fqdn -o tsv)

sed "s/host: smart-router/host: $ROUTER_FQDN/" kong/kong.yml > /tmp/kong-prod.yml
az storage file upload \
  --account-name $SA --share-name kong-config \
  --source /tmp/kong-prod.yml --path kong.yml

SA_KEY=$(az storage account keys list --account-name $SA --query [0].value -o tsv)

# Deploy Kong
az containerapp create \
  --name kong-gateway \
  --resource-group $RG \
  --environment $ENV \
  --image kong:3.9 \
  --target-port 8000 \
  --ingress external \         # public — this is what clients call
  --min-replicas 1 \
  --max-replicas 5 \
  --cpu 0.5 --memory 1.0Gi \
  --env-vars \
    KONG_DATABASE=off \
    KONG_DECLARATIVE_CONFIG=/kong/declarative/kong.yml \
    KONG_PROXY_LISTEN="0.0.0.0:8000" \
    KONG_ADMIN_LISTEN="127.0.0.1:8001" \
    KONG_LOG_LEVEL=info \
    KONG_PLUGINS=bundled \
  --storage-name kong-config-vol \
  --azure-file-volume-account-name $SA \
  --azure-file-volume-account-key $SA_KEY \
  --azure-file-volume-share-name kong-config \
  --azure-file-volume-mount-path /kong/declarative
6

Deploy the Frontend

# Get Kong's public URL
KONG_URL=$(az containerapp show --name kong-gateway --resource-group $RG \
  --query properties.configuration.ingress.fqdn -o tsv)

# Patch the frontend to point at Kong
sed "s|http://localhost:8000|https://$KONG_URL|g" frontend/index.html > /tmp/index-prod.html

# Build a custom nginx image with the patched file
cat > /tmp/Dockerfile.frontend <<EOF
FROM nginx:alpine
COPY index-prod.html /usr/share/nginx/html/index.html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EOF
az acr build --registry $ACR --image frontend:latest \
  -f /tmp/Dockerfile.frontend ./frontend

az containerapp create \
  --name smart-cost-ui \
  --resource-group $RG \
  --environment $ENV \
  --image ${ACR}.azurecr.io/frontend:latest \
  --registry-server ${ACR}.azurecr.io \
  --registry-username $ACR_LOGIN \
  --registry-password $ACR_PASS \
  --target-port 80 \
  --ingress external \
  --min-replicas 0 --max-replicas 2 \
  --cpu 0.25 --memory 0.5Gi
7

Verify and monitor

# Test Kong end-to-end

Verdict

Which one should you pick?

Choose Fly.io if...

This is your personal $40/month LLM setup and you want to spend $10/month on infrastructure, not $35. Fly's free tier can run the whole stack with light usage. Setup takes 20 minutes, maintenance is near-zero, and the developer experience is excellent. This is the right choice for this specific use case.

Also good for: side projects, indie SaaS under 10K req/day, anyone who wants to move fast without learning Kubernetes or Azure IAM.

Choose Azure if...

You're building this for a team or company where the $40/month subscriptions are just the start. Azure gives you Key Vault for proper secret governance, Log Analytics for compliance audit logs, VNet isolation, managed identity (no credentials in env vars), and SLAs you can put in a contract. The extra $20–30/month is worth it at team scale.

Also good for: regulated industries (HIPAA, PCI), teams already in the Azure ecosystem, setups that need AD/SSO integration.

Migration path: start on Fly.io. If usage grows or you onboard a team, the Kong config (kong.yml) is completely portable — copy it to Azure and it works identically. The only migration work is re-deploying the three containers and updating DNS.
